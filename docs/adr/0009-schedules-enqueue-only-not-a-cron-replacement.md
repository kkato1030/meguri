# ADR-0009: 時刻駆動スケジュールは「キューへの起票」だけに限定する — cron の置き換えではない

- Status: Accepted
- Date: 2026-07-13
- Issue: #146

## 文脈

meguri はキュー(ラベル付き issue / ローカルタスク)を消化するループであり、時刻駆動でタスクを生み出す入口がなかった。日次の生産タスクや週次の棚卸しといった定期運用を meguri に載せるには、外部の cron / launchd から `gh issue create` や `meguri add` を叩く別系統のスケジューラを自前で組むしかなく、状態(最終発火時刻)も重複ガードも meguri の外に漏れていた。

`[[projects.schedules]]`(issue #146)で cron 定義から issue/タスクを定期起票できるようにするにあたり、境界をどこに引くかを決める必要がある。時刻駆動の仕組みを一度入れると、「任意コマンドの定期実行」— つまり汎用 cron の置き換え — まで一気に広げたくなる引力が働く。スケジュール定義に `command = "..."` を一つ足すだけに見えるからだ。

## 決定

**スケジュールがやるのは「キューに仕事を積む」ところまで。実行は既存ループの仕事とする。**

- `[[projects.schedules]]` が発火してできるのは、**worker 行き / planner 行きの issue(github mode)または worker 行きのローカルタスク(local mode — planner が無いため v1 は `kind = "ready"` のみ)を1件作る**ことだけである。作られた issue/タスクは、以降まったく通常どおり既存 discovery に拾われて駆動される。スケジューラは起票の入口を1つ増やすだけで、消化パイプラインには一切手を入れない。消化されない起票(local の `plan`)は config 検証で拒否し、「積んだが動かない」状態を作らない。
- **任意コマンドの定期実行(`command = "..."` 的なもの)はスコープ外**とする。ホスト上で任意コマンドを時刻起動したいなら、それは OS の cron / launchd / systemd timer の仕事であって meguri の仕事ではない。meguri は「AI コーディングエージェントをループで回す」装置であり、汎用ジョブスケジューラではない。

この線引きの根拠:

1. **消化側が既に完成している。** worker / planner のループ、claim、エスカレーション、完了、pane 寿命はすべて出来上がっている。スケジューラが `create_issue` / `create_task` を1回呼べば、そこから先は既存の機構がそのまま動く。起票だけに限定することで、新規に作るのは「時刻判定 → 1件作る」という薄い層だけで済む。
2. **Authority 原則(forge が唯一の永続状態)と整合する。** 起票は forge/タスクキューに1行増やす操作であり、状態は既存の場所(issue のラベル、タスクの status)に載る。任意コマンド実行は meguri のライフサイクル外の副作用を生み、この原則の外側に落ちる。
3. **セキュリティ面が単純になる。** 「config に書いた文字列を定期的にシェル実行する」機構は、config を書ける主体に対する任意コード実行の入口になる。起票に限定すれば、スケジューラが持つ権限は「issue/タスクを作る」だけに閉じる。

## 帰結

- スケジュール config のスキーマは起票に必要な項目(`name` / `cron` / `kind` / `title` / 本文)だけを持ち、`command` 相当のフィールドは持たない。将来もこの ADR を覆さない限り足さない。
- 本文の動的生成が欲しくなっても、スケジューラ側では解決しない。capture-first の AI 後追い整形(#120)との組み合わせ、すなわち「まず静的な本文で起票し、その issue/タスクを既存ループが AI で膨らませる」経路で解決するのが筋である。
- 「起票の入口」を増やすこの issue(#146, 1/2)と、「消化の調速」を扱う #148(2/2, discovery の cadence 制御 — not-before / 消化レート上限)は、入口と出口で対になる。両者とも消化パイプラインそのものは置き換えない。
