# ADR 0001: スケジューラの優先度は「パイプラインの逆順」— 新規着手より仕掛かりの完了を優先する

- Status: accepted
- Date: 2026-07-11
- 関連: issue #52, docs/specs/issue-52.md

## 文脈

meguri のパイプラインは planner → (spec review → 人間承認) → worker / spec-worker → reviewer → fixer → merge と流れる。スケジューラのスロット(`max_concurrent`)は有限で、どの仕事にスロットを配るかがシステム全体のスループットと WIP の量を決める。

当初は明示的な優先度がなく、実際の順序は `default_loops()` の登録順と forge が返す issue の順(新しい順)という、いわば偶然の産物だった。結果として新規 issue への着手が仕掛かり品の完了より先に走り、WIP が積み上がりやすかった。

## 決定

**スロットは merge までの残り工程が少ない仕事から配る。** 具体的には、ループのディスパッチ優先度をパイプラインの逆順に固定する:

fixer → spec-worker → reviewer → worker → planner

背後の原則はひとつだけだ:**新規着手より仕掛かりの完了を優先する(WIP を減らす)。** 残工程数はこの原則のよい近似だが、機械的な計数ではない。reviewer(対象の残工程は 5)を worker(残工程 4)より上に置くのは、reviewer が仕掛かり品を人間の承認ゲートまで運ぶ短時間ジョブであり、止めるとゲートごと詰まるからだ。順位が割れたときは残工程の数ではなく原則に従う。

同一ループ内は issue/PR 番号の昇順(FIFO)。古い仕掛かり品ほどコンフリクトのリスクが溜まるので、先に生まれたものを先に完了させる。

優先度の実装は「登録順 + greedy なスロット埋め」だけで表現する。`Loop` trait に `priority()` のような機構は足さない——並び順そのものが仕様である。プリエンプションもしない。走行中の run は最後まで走り、空いたスロットを下流が先に取る。

## 帰結

- 空きスロットは必ず下流(完了に近い側)の仕事から埋まり、下流が涸れてから上流に着手する。下流の仕事は上流が新しい成果物を作らない限り有限なので、飢餓は構造的に起きない。
- ループを追加するときは `default_loops()` の正しい位置——そのループの対象が merge からどれだけ遠いか——に差し込む必要がある。登録順は単なるリストではなく優先度宣言である。
- 複数プロジェクト構成では優先度がプロジェクト順より強く効く(ループ→プロジェクトの順で走査する)。あるプロジェクトの上流仕事は、別プロジェクトの下流仕事に道を譲る。
- 特定 issue の優先(例: `meguri:priority` ラベル)が必要になれば、同一ループ内の並び替えとしてこの構造の上に足せる。優先度機構の一般化はそれまで保留する。
