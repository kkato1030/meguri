# ADR-0012: プロジェクト恒常規律はロール別 preamble として turn プロンプトに埋め込む

- Status: Accepted
- Date: 2026-07-14
- Issue: #149

## 文脈

turn プロンプトは「組み込みテンプレート + issue 本文 + 完了契約」で構成される
(`src/turn/prompts.rs` の `write_prompt_file`、各ループが本文を組み立てる)。ここには
**プロジェクト固有の恒常規律を差し込む場所がない**。恒常規律とは、issue ごとではなく
プロジェクト全体に常時かかる指示を指す — 「作業前に必ずこのガードレール文書を読む」
「成果物はこの文体・ペルソナに従う」「この品質基準を満たさないものはコミットしない」など。

既存の回避策は2つとも穴がある。

- **agent CLI の指示ファイル(`CLAUDE.md` / `.claude/rules/`、ADR 0008 の apm 成果物)**:
  claude 以外のプロファイル(codex 等、routing で混在し得る)には別形式でしか届かず、
  「常時読み込み」しかできない。ロール別に出し分けられない。
- **issue 本文に毎回書く**: 起票者(人間・整形 AI・定期起票)全員が転記を強制され、漏れる。

この機構をどう作るかで、いくつか後戻りしにくい判断がある。それをここに残す。

## 決定

### 1. ロール別 preamble を config で持ち、turn プロンプトに**埋め込む**

`[prompts]`(top-level 既定)と `[projects.prompts]`(per-project)に、ロール名 → repo 相対
パスのマップを持つ。turn プロンプト生成時に、そのロールに該当するファイルを worktree から
読み、プロンプト本文の冒頭に埋め込む。

**参照行(「まず `<path>` を読み、従うこと」)ではなく埋め込みを選ぶ理由**:

- この機構の存在理由の半分は「agent CLI 非依存で同じ規律が届く」ことにある。参照行は
  agent が実際にファイルを読むことに依存する — 読み飛ばせば規律は届かない。恒常規律は
  「必ず届く」ことが要件なので、プロンプトに現物を入れて無条件に届ける。
- 埋め込めばプロンプトは自己完結し、worktree 上のファイルの状態に後から依存しない。
- 欠落時の振る舞いも素直になる。埋め込みなら「その preamble を入れずに warn」で済むが、
  参照だと存在しないパスへの参照行という壊れた指示を残してしまう。

代償はプロンプトが重くなること。preamble はあくまで「恒常規律の前文」であり、短く保つのが
前提とする(かさばる常時コンテキストは `CLAUDE.md` の担当、という住み分けをドキュメントで
案内する)。

### 2. 挿入位置は本文の冒頭、完了契約は末尾のまま優先させる

preamble はあくまで前文であり、完了契約(`result.json` の指示)や検証まわりのテンプレート
文言より優先されてはならない。`write_prompt_file` は完了契約をプロンプト末尾に付ける
構造を保つ — 末尾の meguri 所有テキストが最終権威を握る。preamble はその手前、本文の
冒頭に「プロジェクトの恒常規律」として明示ラベル付きで置き、契約に劣後する前文である
ことを枠付けで示す。

### 3. ロール名の語彙は routing と完全一致させ、未知は config validate で弾く

キーは routing の6ロール(`planner` / `worker` / `fixer` / `self-reviewer` / `pr-reviewer`
/ `cleaner`、`src/routing.rs` の `KNOWN_ROLES`)+ 全ロール共通の `all`。routing と同じく、
非推奨の旧ロール名(`spec-reviewer` → `pr-reviewer` 等、`DEPRECATED_ROLE_ALIASES`)は
`canonical_role` で正規化して受け、それ以外の未知キーは config 読み込み時に `bail!` で弾く。
「静かなフォールバックをしない」という routing と同じ原則に揃える。issue #149 の当初記述は
ADR 0003 改訂(issue #167)前の旧ロール名を並べていたが、正典は現行の6ロールである。

正規化は**検証だけでなく解決(注入)でも効かせる**。マップのキーは照合前に必ず
`canonical_role` を通す — そうしないと、旧名 `spec-reviewer` で書いた entry が validate は
通るのに `pr-reviewer` の turn では引けず、「設定したのに注入されない」という静かな取りこぼしに
なる。この共有のため `routing::canonical_role` を `pub` に上げる。

### 4. 合成順は `all` → ロール別、上書きはキー単位

あるロールの実効 preamble は「`all` の preamble」→「そのロールの preamble」の順に、両方を
連結して埋め込む(共通規律が先、ロール固有が後)。top-level と per-project の合成は
**キー単位**で、per-project 側にそのキー(`all` またはロール名)があればそれが top-level の
同キーを上書きし、無ければ top-level に落ちる。キーは canonical 化してから突き合わせるので、
top を旧名・project を canonical 名にしても同じロールとして正しく上書きされる。同一マップ内で
alias と canonical が同じロールに畳まれる併記は、どちらが勝つか曖昧なので検証で弾く。
`language` の「project が勝つ」流儀を map に広げた形で、top-level に `all` の共通ガードレールを
置きつつ per-project でロール別に足す運用を素直にする。

### 5. 欠落は warn + イベント発火で続行(doctor は別途 strict に検出)

設定されたパスが worktree に無い場合は、warn ログ + イベント発火で該当 preamble を
飛ばして続行する(`worktree_setup` の非 `required` 時と同じ既定)。turn を落とさない。
一方 `meguri doctor` は primary clone 上の存在を strict に検査し、設定したのに無いパスを
問題として報告する — 実行時は「死なない」、doctor は「設定ミスを捕まえる」の役割分担。

### 6. パスは repo 相対に限定する(worktree 外へ出さない)

preamble の中身は agent プロンプトにそのまま埋め込まれる。ここに絶対パスや `..` を許すと、
worktree の外(例: `~/.ssh/id_rsa`)を読んで agent に渡す秘密情報の漏洩口になる。だから
値は「repo 相対パス」を契約とし、絶対パスと親ディレクトリ参照(`..`)を config 検証で弾く。
検証は共通ヘルパに集約し、解決も doctor も読み込む直前に同じヘルパを通してから `join` する
— 経路が増えても関門は一つに保ち、片方だけ検証が抜ける事故を防ぐ。

ただし字面の検証だけでは封じ込めは完成しない。repo 内に置かれた symlink が worktree 外を
指していれば、字面上は正しい repo 相対パスのまま外のファイルへ解決される。よって読み込みは
二段目の関門を必ず通す — `join` した結果と worktree ルートの両方を canonicalize し、実体が
worktree 配下に残っている場合だけ読む(worktree パスの比較を canonicalize で揃えるのは
reaper 等の既存流儀)。外へ出るものは読まずに欠落と同じ warn + イベントで飛ばす。
「config 検証 = 設定ミスの早期拒否、読み込み時 canonicalize = 実体の封じ込め」の二段構えで、
どちらか一方に依存しない。埋め込み方式(決定1)を選んだ以上、この封じ込めは付随的な注意では
なく必須の防御線である。

## 影響

- config スキーマに任意の2セクションが増えるだけで、既存 config はそのまま読める(後方
  互換、DB スキーマ・永続状態には無関係)。ロールバックはセクション削除で完結する。
- #137–#139(agent 指示基盤 / apm)とは補完関係で、置き換えではない。あちらは「worktree に
  指示ファイルを生成・配置する」側、これは「turn プロンプトで何をロール別に読み守るかを
  注入する」側。apm で生成したファイルを preamble に指定する合流も自然。
