# ADR 0005: issue ラベルは「フェーズ × ボールの所在」の2軸モデルにする

## ステータス

採用(issue #102)

## コンテキスト

meguri の issue ラベル遷移では「無ラベル」が3つの全く違う状態を兼ねていて、issue 一覧から
状況が読み取れなかった:

1. **未トリアージ** — 人間がやるか AI に投げるか未決の issue。
2. **実装 PR が open** — worker / planner が PR 作成時にトリガーラベル(`ready` / `plan`)を
   剥がしていた(`src/engine/worker.rs` の `settle_labels`、`src/engine/planner.rs` の
   `settle_labels`)。一番進んでいる状態なのに、issue 上は未トリアージと見分けがつかない。
3. **実装完了後の PR** — spec-worker が着手すると issue 側のラベルが消え、CI 修正・AI レビュー・
   マージ待ちのどこにいるのかラベルからはわからない。

一方で既存の `working` / `needs-human` / `hold` は、トリガーラベルに**重ねて**付く直交マーカー
としてすでに機能していた(`working` はクレーム、`needs-human` は escalate 時に足され `working`
だけを剥がす — `src/engine/flow.rs` の `escalate_on_forge`)。つまりラベル体系は暗黙のうちに
「進行状態」と「誰のボールか」という2軸を持ちかけていたのに、進行状態の軸が PR 作成で欠落して
いた。

## 決定

issue ラベルを明示的に **2軸** で構成する。

### 軸1: フェーズラベル — meguri が関与を始めたら issue クローズまで常にちょうど1つ

| ラベル | 意味 | 色 |
|---|---|---|
| `meguri:plan` | spec 検討待ち(人間が付与) | 🔵 `#1D76DB` |
| `meguri:speccing` | **新設**: spec PR が open(reviewing / ready の詳細は PR 側を見る) | 🟣 `#6F42C1` |
| `meguri:ready` | 実装待ち(人間が付与、または spec 承認後) | 🔵 `#1D76DB` |
| `meguri:implementing` | **新設**: 実装 PR が open(CI 修正・レビュー・マージ待ちを含む) | 🟢 `#0E8A16` |

### 軸2: ボールの所在ラベル — フェーズに重ねて付く(全部なし = ループのポーリング待ち)

| ラベル | 意味 | 色 |
|---|---|---|
| `meguri:working` | AI がいままさに作業中(クレーム) | 🟡 `#FBCA04` |
| `meguri:needs-human` | 人間のボール(理由はコメント) | 🔴 `#B60205` |
| `meguri:hold` | 人間が意図的に停止 | ⚪ `#CFD3D7` |

`needs-human` はフェーズラベルを**剥がさず重ねる**。「spec で詰まった」のか「実装で詰まった」の
かがフェーズラベルの色に残る。これは既存の `escalate_on_forge`(`working` だけ剥がす)の挙動を
そのまま2軸モデルの規約として明文化したものであり、実装の変更を伴わない。

### PR 側は現状維持

spec PR は `meguri:spec-reviewing` / `meguri:spec-ready` + マーカーのまま。CI 赤やマージ可否は
GitHub がネイティブに表示するので、ラベルへのミラーは同期の複雑さの割に得るものが少ない。
必要になったら `meguri:awaiting-merge` を後から足せる設計にしておく(フェーズラベルは issue 側、
PR 側ラベルとは独立という切り分けが、この拡張余地を担保する)。

## この決定が確立するドメイン不変条件

- **meguri が関与した open issue には、フェーズラベルがちょうど1つ付いている。**
  したがって **issue の無ラベル = 未トリアージ(人間が判断する)** という一義的な意味になる。
  例外は planner の decompose で子に分割された親 issue で、これは全フェーズラベルを外れて
  「人間が最終判断する」状態に戻る(コメントで経緯を残す)— 無ラベル = 人間の判断待ち、という
  上の意味と整合する。
- ボールの所在ラベルはフェーズラベルに**重ねて**のみ付く。フェーズラベルを剥がさない。
- 🔴 `needs-human` だけをフィルタすれば、人間の TODO 一覧になる。

## 帰結

- issue 一覧が状態機械のスナップショットになる: 色でフェーズ(spec 検討中=紫 / 実装中=緑 /
  待ち=青)、赤で人間のボールが一目でわかる。
- ラベル遷移の実装は「剥がして無ラベルにする」から「フェーズラベルを差し替える」に変わる
  (worker / planner / spec-worker の settle 経路)。ボールの所在ラベルの扱いは無変更。
- cleaner は不変条件の観測点になる: フェーズラベルが 0 個または 2 個以上の open issue を
  乖離として報告する(書き込み境界はレポート issue のまま — ADR 0003)。
- ラベル色が意味を担うので、色は `ensure_label` にコード化して収束させる(スキームが wiki 上の
  約束事に留まってドリフトするのを防ぐ)。
- 将来 PR 側にマージ待ちの可視化が要るなら `meguri:awaiting-merge` を PR ラベルとして足せる。
  フェーズラベル(issue 側)とは軸が違うので、本 ADR の2軸モデルを壊さない。
