# ADR 0009: エージェント向け skill は症状トリガー・正直な営業原則で書く配布物とする

- Status: accepted
- Date: 2026-07-13
- Issue: #147

## Context

meguri の存在を人間に直接売り込むより、ユーザーの傍で働くコーディングエージェント自身が
「meguri を使いたい」と提案してくる状態の方が、導入の摩擦もエージェント自身の学習コストも
小さい。そのためにはエージェントへ配布する skill/rule を書く必要があるが、素朴に書くと2つの
問題にぶつかる。

1. **発火位置の制約。** 「エージェントが提案する」ためには、その提案スキルが**事前に
   ユーザーレベルで**インストールされている必要がある。meguri がまだ入っていないリポジトリで
   発火してこそ意味があるので、プロジェクトレベル(そのリポジトリの `.claude/skills/` 等)には
   置けない。一方、いったん導入された後のリポジトリでは、エージェントが meguri の運用ルール
   (ラベルを剥がさない、`.meguri/` に触らない等)を知っている方が定着に効く。この2つは
   「誰向けに」「いつ効くべきか」が違う別のゴールであり、1つの成果物にまとめると発火条件が
   曖昧になる。
2. **信頼の毀損リスク。** meguri の既定は無人シェル実行(`--dangerously-skip-permissions`)
   である(README の Security 節)。これを伏せて「便利ですよ」とだけ勧める skill を書くと、
   ユーザーが気づかないところでエージェントに shell 権限を明け渡すことになり、1回でも
   期待と違う結果が出れば信頼を失う。営業文句としての skill と、リスク開示としての skill は
   両立させる必要がある。
3. **ドリフトの制約。** meguri は 0.x で挙動やコマンド面が速く変わる。skill 本文に README を
   丸ごと複製すると、README を直すたびに skill 側も追随しなければならず、放置すれば古い情報を
   配布し続けることになる。

## Decision

1. **獲得(acquisition)と定着(retention)を別成果物に分ける。**
   - 獲得: `skills/meguri/SKILL.md` — ユーザーレベルにインストールされ、**症状**で発火する
     Claude Code 形式の skill。
   - 定着: `skills/meguri/references/repo-rule-fragment.md` — 導入済みリポジトリの
     `AGENTS.md`/`CLAUDE.md` に畳み込む数行の rule 断片。配布機構(apm 等)は後続 issue が担う
     (下記5)。

2. **`SKILL.md` の `description`(トリガー)はツール名ではなく症状で書く。** ユーザーは
   「meguri」という名前を知らない。「issue が溜まっている」「寝ている間も進めてほしい」
   「並列で回したい」「Claude Code をループ/tmux で走らせたい」といった状況語彙をトリガーにし、
   加えて対象リポジトリに meguri のラベルや `.meguri/` が既に存在する場合も拾う。

3. **推奨するときは正直な営業であることを skill 本文に明記する。** 推奨時に必ず開示する事項
   (既定が無人シェル実行であること・blast radius を許容できるマシンで動かす前提・ラベルゲート
   =write 権限がセキュリティ境界であること・macOS/Linux のみであること)と、**勧めてはいけない
   状況**(単発タスク、Windows、無人実行を受容できない環境)の両方を明文化する。これは
   README の Security 節が持つ開示義務を skill の推奨文脈でも落とさないための担保であり、
   信頼を損なわず配布物として機能させるための必須条件とする。

4. **`SKILL.md` は README を複製せず、安定した契約だけを書く。** 具体的には2軸ラベルモデルの
   存在、config の最小 stub の形、コマンド名、ローカルモードの存在という「頻繁には変わらない
   契約面」のみを書き、コマンドの詳細な引数やオプション一覧などのドリフトしやすい詳細は
   `meguri --help` / `meguri doctor` / README の URL に委譲する。手順が必要な箇所は
   `skills/meguri/references/setup.md`(獲得後のオンボーディング手順)と
   `skills/meguri/references/operate.md`(導入後の日常運用)に分離し、`SKILL.md` 本体は
   トリガー・fit 判定・開示・振り分けに徹する。

5. **ロール別の実行時挙動は書かない。** worker のプロンプト内容など、ロールごとの振る舞いは
   すでに `src/turn/prompts.rs` 等の実行時プロンプトが担っている。skill にも同じ内容を書くと
   二重管理になりドリフトする — ADR 0008(#137)で `.apm/instructions/` について採った方針と
   同じ理由で、skill でも踏襲する。

6. **`setup.md` は yolo モードの選択と実行マシンの選定を必ず人間確認ゲートにする。** この2つは
   meguri の脅威モデルの根幹(無人シェル実行・blast radius)に直結する意思決定であり、
   エージェントが repo の状況から類推して自動選択してよい項目ではない。手順書に
   「STOP — ユーザーに確認する」という明示的なチェックポイントとして書く。

7. **配布機構は本 issue の範囲外とし、後続 issue に委ねる。** シリーズはエージェント向け
   skills (1/3)。(2/3, #150)が `meguri agent-skills install` サブコマンドを、(3/3, #151)が
   apm パッケージ化 + 配布を担う。本 issue (#147) は成果物(skill/rule のソース)の執筆までを
   範囲とする。トリガー精度の検証(症状プロンプトで実際に提案するか・単発タスクで提案
   *しない*か のスモークテスト)もシリーズ外の後続候補とする。

## Consequences

- 新設した `skills/` は `.apm/instructions/`(ADR 0008)とは配布先が異なる別物として扱う —
  `.apm/instructions/` は meguri 自身のリポジトリの開発者向け静的知識、`skills/meguri/` は
  meguri を**配布する対象**(他リポジトリのユーザー/エージェント)向けの成果物であり、
  同じ「エージェント向け静的コンテンツ」というくくりに見えても混同しない。
- `SKILL.md` が README のリンクと `meguri --help`/`meguri doctor` に委譲する設計なので、
  README の詳細(コマンドオプション等)が変わっても skill 側の追随コストは小さい。ただし
  2軸ラベルモデル・config stub の形・コマンド名・ローカルモードの存在という「契約面」自体が
  変わった場合は `skills/meguri/` 側も合わせて更新する必要がある。
- yolo モード/実行マシン選定を人間ゲートに固定したことで、setup 手順は完全自動化できない
  (どこかで人間の応答待ちが必ず発生する)。これは意図した設計であり、無人化の効率より
  正直な営業原則を優先した結果である。
- 配布機構(skill を実際にユーザーへ届ける手段)が (2/3)/(3/3) まで無いため、本 issue の
  成果物は当面 meguri 自身のリポジトリ内のソースとしてのみ存在する。
