# ADR 0008: issue 本文の編集は「再着手シグナル」であって「実行トリガー」ではない

## ステータス

採用(issue #142)

## コンテキスト

reconcile loop が反応するのは「ラベル・open/closed・新規コミット・CI/mergeable 状態・
新規レビュースレッドコメント」であり、**issue 本文(description)の編集**は検知対象外だった。
一度 succeeded した (project, loop, issue) は `issue_has_succeeded_run`
(`src/store/runs.rs:325`)で**恒久的にサプレッション**され、その後に概要欄を編集しても
再ディスカバリされない(`meguri run --issue N` で明示バイパスするしかない)。

この穴を塞ぐにあたり、2 つの設計上の緊張がある:

1. **ラベルが唯一の真実源というモデル**(`src/tasks.rs`、ADR 0005)。github モードでは
   ワークフロー状態のミラー行を持たず、フェーズラベルが状態機械そのものだった。
2. **起動権限モデル / prompt-injection 対策**(`README.md:53`)。issue 本文は agent の
   プロンプトへ verbatim に注入される。label gate によって「誰が agent を起動できるか =
   誰が repo に write 権限を持つか」に還元されており、これが誰でも issue を開ける repo での
   prompt-injection の主要な緩和策になっている。本文編集を実行トリガーにすると、この
   ゲートを迂回して「issue を編集できる誰か」が agent を起動できることになり、権限モデルが緩む。

また GitHub の `updatedAt` は**ラベル付け替えでも動く**ため、「本文が実体として変わったか」を
`updatedAt` だけでは見分けられない。一方で本文(`body`)はディスカバリ時点で既に手元にある
(`list_issues_with_label` が返す)。

## 決定

**issue 本文の編集を、実行トリガーではなく「再着手を促すシグナル」として扱う。**
実行の権限ゲートは従来どおりラベル(collaborator 権限)のままにする。

具体的には:

1. **検知は本文の内容ダイジステで行い、`updatedAt` は使わない。** ディスカバリ時点で
   持っている本文を正規化(前後空白 trim・連続空白の畳み込み・改行 LF 統一)して
   SHA-256 を取り、前回処理時のダイジェストと比較する。ラベル churn は本文を変えないので
   原理的に無視され、追加の API 呼び出しも要らない。

2. **「処理済み」判定を本文ダイジェストに対して行う。** `issue_has_succeeded_run`
   (処理済みなら常に抑止)を、現在の本文ダイジェストが処理時のものと一致するときだけ抑止する
   ように改める。本文が変われば抑止は解除され、再ディスカバリの余地が生まれる。

3. **抑止が解けても、本文編集そのものでは agent を起動しない。** worker が実際に走るのは
   トリガーラベル(`meguri:ready` など、collaborator が付与)が付いているときだけ。success 後の
   issue はフェーズラベルが `implementing` に差し替わっていて `ready` を持たない(ADR 0005)ので、
   本文を編集しても自動起動はしない。reconcile loop は本文変更を検知してイベントと一度きりの
   コメントで人間に「再実装が要るなら `meguri:ready` を付け直して」と促すに留め、実際の再着手は
   collaborator のラベル付け直しで発火する。

## この決定が確立するドメイン不変条件

- **本文編集は agent を起動しない。実行の権限ゲートはラベル(collaborator 権限)である。**
  したがって「誰が agent を起動できるか = 誰が repo に write 権限を持つか」(README の label gate)は
  本文編集検知を足しても維持される。
- **「処理済み」は本文の内容に対して定義される。** ラベルの付け替えでは処理済み判定は動かない。
  本文が実質的に変わったときだけ、恒久サプレッションが解ける。
- **本文ダイジェストはラベルをミラーしない。** これは既存の `issue_has_succeeded_run` と同じ
  ローカルな冪等性補助であって、ワークフロー状態の真実源ではない。ラベル真実源モデル(ADR 0005)は
  そのまま。

## 帰結

- `updatedAt` を forge 取得に足さずに済む。本文比較の方がラベル churn への頑健性が高い。
- サプレッションのダイジェスト化はレガシー互換で入れる: ダイジェスト NULL の既存 succeeded run は
  「常に一致」= 従来どおり恒久抑止として扱い、アップグレード時の暴発を防ぐ。ダイジェストを記録するのは
  本 issue 以降の新しい success だけ。
- 検知と再着手の遷移は `issue.body_changed` イベントで durable に残り、`meguri logs` / events から
  追える。
- 将来 `[reconcile] body_edits = "auto"` として「本文編集で自動再着手」を opt-in できる設計余地は残す。
  ただしデフォルトは signal-only で、上の権限不変条件を守るのは opt-in しない限り変わらない。
- 既存コメントの編集やコメントの自然文 trigger は本 ADR の射程外(別 issue)。本文編集だけを
  シグナル化することで、prompt-injection の面を広げずに恒久サプレッションの穴を塞ぐ。
