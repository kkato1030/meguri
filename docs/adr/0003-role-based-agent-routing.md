# ADR 0003: エージェントの振り分けは役割ベース — 明示は auto に勝ち、失敗は起動時に大きな音を立てる

- Status: accepted
- Date: 2026-07-12
- Issue: #64
- 関連: docs/specs/issue-64.md(routing 1/3。後段: 2/3 成果ベース検査、3/3 エスカレーション)

## 文脈

meguri のループ(planner / reviewer / worker / spec-worker / fixer / conflict-resolver)は、トークン消費と品質の重みのプロファイルがきれいに分かれている。planner は消費が小さいのに spec の質が下流のターン数すべてを左右し、worker は消費の 8〜9 割を占め、fixer は狭いスコープの小 diff を作るだけだ。全ループが同じモデル・同じ CLI で走る必然はなく、振り分けはコスト(サブスクリプション枠の消費)と品質の両方に効く。

振り分けの軸には「issue の難易度を推定して割り当てる」という案もあった。

## 決定

**振り分けは役割(loop_kind)ベースにする。難易度推定はしない。** 難易度推定はそれ自体に LLM 呼び出しが要る上に外れるが、役割は run が生まれた時点で確定している決定的な軸で、テストも検証も容易だ。

その上で 2 つの運用原則を固定する:

1. **明示設定は常に auto に勝つ。** `[routing.roles]` に書かれた役割は必ずそのプロファイルを使う。プロファイルが存在しない・CLI が検出できない・役割名が未知なら、**起動時に明示エラー**で止まる。静かに別のプロファイルへフォールバックすることはしない(daemon の非対応プラットフォーム処理と同じ流儀)。ユーザーが書いたものと違うものが黙って走るのは、動かないことより悪い。
2. **auto の推奨表はバイナリに焼き込み、生成日(`GENERATED_AT`)を持つ。** 推奨は「2026-07 時点の相場観」であって普遍の真理ではない。モデルの世代交代で必ず古びるので、データとして日付を持たせ、鮮度の検査(routing 2/3)を機械的に行えるようにする。auto は推奨チェーンを実行環境の検出(`command --version`)でフィルタして適用し、チェーンの終端は常に `default`(従来の `[agent]`)に落ちる。

後方互換は「`[routing]` を書かなければ何も起きない」で担保する。auto の発動条件は `[routing]` セクションの明示に限定し、`[agents.profiles]` の定義だけでは推奨表は発動しない — profiles は `[routing]` から参照されるまで完全に不活性(inert)で、全役割が従来どおり `[agent]` で起動する。これは原則 1 の裏返しでもある: セクションの存在だけがスイッチになってユーザーが書いていない振り分けが黙って走る状態を作らない。

## 帰結

- ループを追加するときは、そのループの役割を推奨表(チェーンと根拠)に登録する必要がある。役割の追加 = ルーティング表の 1 行であり、ルーティング機構そのものには手が入らない。
- 明示設定のエラーは起動時(`meguri watch` / `meguri run` の入口)に集中する。走行中の run が設定不整合で別プロファイルにすり替わることはない — run 開始時に固定したプロファイル名(`runs.agent_profile`)が消えていたら、それも明示エラーになる。
- 推奨表の更新は「表の書き換え + `GENERATED_AT` の更新」という純粋なデータ変更になり、レビューで根拠(モデルの単価・品質の相場)を問える。
- 難易度ベースの振り分けが将来必要になっても、役割ベースの解決の**下**に足す(同一役割内での選択)。役割という決定的な軸を捨てる理由にはならない。
